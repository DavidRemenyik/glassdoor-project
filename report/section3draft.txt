Section 3. MapReduce Optimisation3.1 Identification of the Bottleneck and ExpectationsIn our traditional, single-threaded prototype (Section 2), the most time-consuming bottleneck was the data parsing and mathematical transformation phase. For every single row in the massive Glassdoor dataset, the system had to extract five distinct string columns, cast them to floating-point numbers, handle potential missing values ("NULL"), and apply a complex custom weighting formula (e.g., work_life_balance * 1.1197 + culture_values * 1.4247...). This row-by-row CPU-bound operation resulted in an execution time of approximately 1 minute and 51 seconds.By transitioning this workload to a big data programming paradigm using the Spark Core API (RDDs), our expectation was to reduce the execution time by at least half. We anticipated that applying the MapReduce paradigm would allow us to distribute the heavy row-parsing and mathematical weighting across multiple CPU cores simultaneously.3.2 MapReduce Solution (Spark Core API)To strictly adhere to the MapReduce programming paradigm, we utilized the Spark Core API (Resilient Distributed Datasets) rather than high-level DataFrames. Our logic was decomposed into the following Map and Reduce functions:The Map Function: We implemented a narrow dependency transformation using mapPartitions. Instead of calling a Python function for every single line, mapPartitions passes whole chunks of the CSV to the worker nodes, drastically reducing function-call overhead. The Map function drops the CSV header, filters out rows containing "NULL" in the target columns, parses the floats, calculates the final weighted score, and emits a Key-Value pair for each valid review.Key: The Firm's Name (e.g., "Google").Value: A tuple containing a count of 1 and the calculated score: (1, weighted_score).The Reduce Function: We applied reduceByKey, a wide dependency transformation that shuffles data across the cluster. For each identical Key (Firm), the Reduce function aggregates the Values by summing both the counts and the scores: lambda a, b: (a[0] + b[0], a[1] + b[1]).Post-Processing: Finally, we filtered the reduced RDD to only include firms with $\ge 1000$ reviews and mapped the values to calculate the final average (total_score / count), before sorting the results.3.3 MapReduce Results and Performance AnalysisOur MapReduce implementation successfully output the exact same Top 10 Weighted Companies (led by Continental-Software-Solutions and Universitas-Gadjah-Mada) as our traditional SQL approach, validating our logic.Execution Times:Traditional SQL Baseline: ~111 seconds (1m 51s)Basic Spark RDD (CSV): ~55 secondsOptimized Spark RDD (CSV): ~42 secondsSpark RDD (Parquet format): ~130 seconds (2m 10s)Analysis of Deviations and Expectations:The transition to MapReduce successfully matched our expectations, cutting the execution time from 111 seconds down to 42 seconds. We achieved this optimized 42-second runtime by explicitly configuring the SparkContext to utilize all available local cores (local[*]) and replacing the default Java serializer with the highly efficient KryoSerializer to speed up the reduceByKey shuffle phase.However, we encountered a highly insightful deviation when experimenting with data storage formats. We hypothesized that migrating our dataset from CSV to a columnar Parquet format would further reduce the MapReduce execution time. Unexpectedly, reading Parquet files into the Spark Core RDD API resulted in the slowest execution time of all (2 minutes 10 seconds). This severe deviation is due to the "PySpark Serialization Tax." While Parquet is highly optimized for Spark SQL running natively inside the Java Virtual Machine (JVM), converting that columnar data into an RDD requires Spark to deserialize millions of records in the JVM and push them across the Py4J bridge into Python object space so our custom map function could read them.Consequently, we concluded that when strictly utilizing the low-level Spark Core API and custom Python Map functions, keeping the data in flat CSV files and relying on mapPartitions provides vastly superior throughput compared to columnar formats.
