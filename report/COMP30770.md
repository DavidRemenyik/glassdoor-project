**Section 1: Introduction**

The dataset we chose consisted of two Kaggle datasets containing company reviews, which we combined into a single dataset of 10.7 million reviews, an \~1.5GB CSV file. 

The dataset qualifies as big data due to its volume, variety and velocity. 

**Volume:** 10.7 million records exceeds the memory capacity of standard Python/Pandas applications. 

**Variety:** The dataset is mixed-mode data. It contains structured numerical ratings (e.g scores for work-life balance), categorical data (e.g company names), unstructured text fields (e.g employee “Pros” and “Cons”). 

**Velocity:** Glassdoor review datasets will grow continuously as employees submit new feedback from their employment. Our pipeline is designed to be “iterative” to accommodate for new data to be taken in and analyzed as the dataset grows. 

The overall objective of this project was to identify the top-rated companies by analyzing employee reviews and creating a custom weighted sentiment model. This allows us to determine the “true value” of companies based on employee sentiment, rather than relying solely on traditional star ratings.

## **Feature Engineering (The Weighting Algorithm)**

The original system based the company rating on 6 different categories, each with a 1-5 star scale, and the final value was just a simple average of those values. We saw this as an opportunity to discover a more accurate rating solution.

The idea was to get a better understanding of the employees choices on a deeper level, as personal morals and experience can cause bias in a review. For example someone with a previous bad experience with senior management may take more care to scrutinise senior management, leading to a less accurate representation of the overall value.

* The Algorithm: For every review, we scanned the "Pros" and "Cons" for specific buzzwords relating to each category (e.g., "overtime," "toxic," "benefits," "flexible").  
* Calculation: We count all “positive” values to be buzzwords mentioned in the “Pros” and vice versa for the negatives.  
  * Score \= (Mentions in Pros) \- (Mentions in Cons)

  We take this value and multiply it by the average rating for that category to give a more accurate value.

* The Discovery: We found that "Culture Values" had a massive total score (1.1M mentions) , seeming that employees put a lot of emphasis on a friendly working environment. While "Senior Management" had a negative score ( \-883k), indicating it is more often cited as a "Con."  
* Final Weights are a percentage score from the category mentions/every mention:  
  * Work-Life Balance: 0.1197  
  * Culture Values: 0.4247  
  * Career Opps: \-0.0129  
  * Comp & Benefits: 0.2584  
  * Senior Mgmt: \-0.3348  
  * Diversity & Inclusion: 0.0052

## **SQL Prototype**

To compute the scores for each company based on our weighted algorithm we implemented a traditional SQL-based solution using MySQL 8.0 running in a docker container. We ingested the dataset into a table in order to be able to use and run MySQL queries on it.

The following query returns the name of each company, the amount of reviews they have and the calculated weighted average score of the company. We filtered that the company must have at least 1,000 reviews to reduce statistical noise.

SELECT firm, \\  
    COUNT(**\***) AS review\_count, \\  
    (AVG(\\  
          work\_life\_balance   **\***  1.1197 \\  
        **\+** culture\_values      **\***  1.4247 \\  
        **\+** (career\_opp          **\+** (career\_opp **\*** **\-**0.0129)) \\  
        **\+** comp\_benefits       **\***  1.2584 \\  
        **\+** (senior\_mgmt         **\+** (senior\_mgmt **\*** **\-**0.3348)) \\  
    ) **/** 5\) AS weighted\_rating \\  
FROM employee \\  
WHERE work\_life\_balance IS NOT NULL \\  
  AND culture\_values IS NOT NULL \\  
  AND career\_opp IS NOT NULL \\  
  AND comp\_benefits IS NOT NULL \\  
  AND senior\_mgmt IS NOT NULL \\  
GROUP BY firm \\  
HAVING COUNT(**\***) **\>=** 1000 \\  
ORDER BY weighted\_rating DESC; \\

**Section 3: MapReduce Optimisation**

## The most computationally expensive step in the traditional solution was computing the ratings for each company using our new weighted algorithm. This operation required scanning the full dataset and performing large-scale grouping and averaging. This computation can be partitioned across multiple machines so it is well suited to MapReduce.

We expected Spark to reduce execution time by approximately two to four times compared to the traditional solution using SQL. 

### **Why Resilient Distributed Dataset Core API?**

Per the project brief, we utilized the Spark Core API (RDD) rather than DataFrames. This allowed us to explicitly define the Map and Reduce phases as follows:

1. Fault Tolerance: We utilized RDD Lineage. If a node fails while processing the 1.5GB file, Spark uses the lineage graph to recompute only the missing partition.  
2. The "Serialization Tax": We discovered that moving 10.7M Python objects to the JVM is slow. To optimize this, we implemented:  
   * mapPartitions: Instead of calling a function 10.7 million times, we call it once per partition (\~100 times), passing a batch of rows.  
   * KryoSerializer: Replaced the default Java serializer for faster shuffle performance.  
   * csv.reader: Used the Python csv module inside the Map phase for robust parsing of text fields.

### **Native Performance (Scala RDD)**

To further investigate the "Serialization Tax," we implemented a native Scala RDD version (spark/GlassdoorAnalysis.scala).

* Startup Overhead: Because we used the spark-shell to run the Scala code, the initialization of the JVM and Spark environment adds significant overhead (\~35-40 seconds).  
* Processing Speed: Once initialized, the Scala implementation processed the 10.7M rows in just 24.8 seconds.  
* Advantage: By running directly on the JVM, we completely bypassed the Python-to-Java communication layer (Py4J). This version achieved the highest throughput for raw RDD processing on CSV data.

## **Beyond the Paradigm (DataFrames & Parquet)**

As an extension of our research, we explored more modern Spark features to see how far we could push the performance.

### 

### **DataFrames vs. RDDs**

We implemented the same logic using Spark DataFrames (sql-query-spark.py). DataFrames use the Catalyst Optimizer to generate an efficient execution plan.

* DataFrame (CSV): While faster than MySQL, it still suffered from parsing overhead, taking 1 minute 41 seconds.

### **The Parquet Optimization**

We converted our 1.5GB CSV into Apache Parquet, a columnar storage format.

* DataFrame (Parquet): The execution time dropped to a staggering 16 seconds. This is because Parquet only reads the specific columns needed for our formula (work-life balance, Culture, etc.), skipping the heavy text of "Pros" and "Cons."  
* RDD (Parquet): We attempted to read Parquet directly into an RDD (rdd-parquet-spark.py). This was actually slower (2m 10s) than the CSV-based RDD. This taught us that the RDD Core API is not optimized for complex columnar formats; the conversion from Parquet's structured schema back to a flat Python RDD creates more overhead than simply reading a raw text file.

### **Final Discovery Conclusion**

Our journey through the Big Data stack confirms that while MapReduce (RDD) provides the most control and transparency for custom algorithms, the choice of Language and Storage Format are the two biggest performance levers:

1. Language: Moving from Python to native Scala halved the RDD processing time from 55s to 25s by eliminating the Py4J serialization tax.  
2. Storage: Converting to Apache Parquet dropped the DataFrame time to a staggering 16 seconds by leveraging columnar storage.

For the purposes of the foundational MapReduce paradigm as taught in COMP30770, our Optimized Scala RDD (25s) represents the most efficient use of the core API on raw text data, proving that JVM-native execution is critical for high-performance Big Data pipelines.
